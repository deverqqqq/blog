Static Site vs Dynamic Site

  1. 什麼是靜態網站( Static Site )?

    A static website contains Web pages with fixed content.

    Each page is coded in HTML and displays the same information to every visitor.

    來源

    個人解讀：靜態網站就像是一本實體書，你把這本實體書放在任何人的面前，每個人看到的內容都會是一樣的。

    (網站與使用者沒有互動，僅提供內容呈現)

  2. 什麼是動態網站( Dynamic Site )

    Dynamic websites contain Web pages that are generated in real-time.

    來源

    個人解讀：動態網站就像是一個電子閱讀器，會根據使用這台機器的登入者，判斷你可以看到那些內容。

    (網站與使用者是有互動的，不只是單方面提供內容)

  3. 動態網站 | 靜態網站 哪個好 ( ex: Blog )?

    • 取決於你的用途

      以 Blog 為例，最基本的事情就是閱讀，從這個角度看，的確很適合用靜態網站來建置，畢竟大多數的人，應該不會想要做得太複雜，況且 Blog 的本意，本來就是記錄自己的文章，需要根據 ( 會員 / 群組 ) 來判斷可以看到那些文章，反而比較像是討論區的功能。

    • 哪些功能需要透過動態方式實現

      顯而易見的，【留言】就是一個需要動態支援的功能，試想，你把10本書賣出去後，其中一名顧客在書上塗鴉，另外九本書難道也會顯示這些一樣的塗鴉嗎? 這是靜態網站所做不到的；但如果是動態網站，我就可以在每一頁都新增一個留言區塊，當有人閱讀某一頁，想要留言的時候，只要在區塊內留言並送出，儲存在後端的資料庫內，下次有其他讀者看到這一頁，就知道原來之前有人，看到這一頁留下了什麼訊息。

    • 哪一個比較簡單

      以自行架設來說，現在網路上大多都有懶人包了，其實有心接觸的話，做出一個好用穩定的網站都不會太難，但如果考慮到後期的維運，要考慮和注意的事情，我覺得靜態網站是比動態網站還要簡單的。

  4. 我只想紀錄東西，有這麼難嗎?

    使用既有平台的服務是最簡單可靠的，像是 Blogger 或是 Medium，基本上只要登入後透過 UI 操作一下，馬上就可以有可靠又好看的網站了，而且因為平台使用者多，想要自訂主題，新增外掛功能，大致上都找的到相對應的解決辦法，是一個非常推薦給不想要自己折騰的人使用。

不管是否要自己架設網站，架設的網站是採用靜態網站還是動態網站，最重要的還是內容跟內容的保存、管理。

站在讀者角度：沒有人會看一個空無一物的網站；站在作者角度：沒有人會想讓自己的心血，化為烏有。

確保了這兩件事情，再來慢慢考慮其他的功能，我覺得才是正確的做法。

現有 Blog 服務

  1. Blogger

    非常有歷史的的 Blog 寫作平台，背後是 Google 的伺服器，在與自家服務的整合上面 ( Google Drive )，或是連線的穩定和速度，都是值得推薦的。

    對於單純想要試試看寫 Blog 的人來說，我會推薦使用這個平台，簡單、穩定、好上手、變化度高。

    PS: Blogger 主題網站，之前自己玩 Blogger 常用的主題網站，通常是從這邊下載後，再做後續的調整。

  2. Medium

    這幾年算是滿流行的一個新的 Blog 平台，主打專業有深度的文章發布平台，整體上給人的感覺比較像是，吸引作者建立有內涵的內容，像一個社群的概念，久而久之，會有越來越多的人依賴 Medium 找到其他有深度但是不同主題的文章。

    不過因為我自己並沒有使用過，有興趣的人可以自己嘗試來玩玩。

  3. 其他

    蒐集資料的過程，才發現原來現在還有這麼多有趣的平台，可以參考下面的連結做一個基本的認識。

選擇建置 Staic Site 的原因

👨‍💻完整掌控自己的內容

對於沒有網站基本知識的人，或者是只是想要單純寫作的人，也許會覺得很抽象，甚至是覺得沒必要搞那麼麻煩但對於真正想要長久經營的人，嘗試從另外的層面，來思索這個問題，應該會有意想不到的發現。

  1. 你的網站不是你的網站

    說白的就是，你不知道你選擇的平台，之後會不會收掉，又或者是改變營運的模式，如果你每天的熱情地寫章，結果三年後，平台老闆表示，因為某種原因，我們將於半年後停止服務。 這時候你面對的，不僅僅是要在有限的時間內，找到另一個可靠的平台繼續創作自己的內容，而且因為平台不同，匯出的檔案格式可能也不相同，如果缺少相對應的工具，要轉過去也是很困難的一件事情，更不用提主題這些了，不同的平台間的主題不可能完全相同，自己在 A 平台修改好的主題，到 B 平台有很大機率不用 / 不能完全相容。

    👉使用 Static Site，不管是放在哪一個主機，呈現的結果都是一樣的，所有渲染後的檔案都已經產生了， Server 端唯一要做的只是把檔案給訪客下載。

    ( 嚴格說起來不一定，但大致上可以這麼理解 )

    當我們在寫文章的時候，實際上，隱藏了很多資訊在背後，例如：建立時間、最後修改時間、作者、標籤、分類…等。當我們從平台下載備份的時候，必須將這些相關資訊匯出，才能到新的平台匯入使用，但是新平台的文章格式許跟舊平台不完全一樣，說不定只有標籤，沒有分類，也可能只有分類，沒有標籤，各式各樣的狀況都有可出現，這些規格都是由各平台自己主導的，如果缺了轉換的工具，想要直接匯入是辦不到的事情，更別提搬家後的圖片，可能會有連結失效要重新調整的問題。

    轉換的工具其實不難找到，但萬一有一部分文章資訊沒辦法完整轉換，這樣層出不窮的小問題，就會讓人心意亂。

    此外，有沒有發現，當我把文章下載來之後，竟然不是人類可以越閱讀的格式，而是給程式閱讀的格式，辛苦撰寫的筆記、文件、生活點滴…等，下載後竟然無法在自己的電腦上面舒服的閱讀，一定要靠網站渲染才可以閱讀自己的文章，對我來說根本是本末倒置。

    👉使用 Static Site Generator + Theme 設定，可以把 Markdown 文件，渲染成好看的網站，自己只要好好專心維護 Markdown 文件就可以了。

    PS: Markdown 文件一種非常好編輯、好閱讀的語法

  3. 你的主機不是你的主機

    當你看到這邊，也許想說，那我用 WordPress 架站，從頭到尾就都是自己的東東嗎?

    沒錯，如果是用 WordPress，找到合適的 VPS，透過社群討論的文章與許多前人的懶人包教學，很快就可以完成一個完整、穩定，你想的到的，你想不到的各種功能插件，擺在那邊給你用，何樂而不為。

    但是，身為一個在資訊產業打滾的外星人，看到很多幾個步驟架站的文章，其實內心有許多的不安

    • 你知道你的主機運行哪一套作業系統嗎?
    • 你知道你的作業系統有沒有自動做安全性更新嗎?
    • 你知道你的網站是定期備份到哪裡去的嗎?
    • 你知道你要如何把備份的資料倒回來嗎?
    • 你知道如何去監測網站的流量是否有異常嗎?
    • …等

    如果上面這些事情你都不知道，也不想知道，那這個主機真的算是你自己擁有的嗎?

    如果是為了學習網站知識 + 寫作，絕對是推薦使用 WordPress 作為架站的工具，這個 Framework 不管是流行程度、活躍度、社群討論數，都稱得上是 T1 的等級。( 與之相伴的，想要攻擊這個框架弱點的人，也會跟著增加，這個跟 Windows 電腦容易被攻擊很像 ( 排除系統本身的原罪 )，當使用的人數越多，就會容易成為 Cracker 的目標 )

    但如果是為了擺脫平台的控制，而忽略這些潛藏風險，至少我自己是不願意的。這也是為什麼我會在前面說到，其實我很推薦既有的平台服務，只要他使用人數夠多，營運的夠長久，要搬家其實也不愁沒有轉換工具 ( 大不了自己做一個 )，就算會有惱人無法完整轉換的問題，對我來說，也比自己去操煩那些主機的事情來的好。

    我相信以現在的懶人包技術，各種安全機制或措施應該大致上都做得相當完善，只是身為工程師多疑的個性，實在很不喜歡把自己暴露在未知的風險之中。

    👉 Static Site 要放在自行購買的 VBS 服務上面，或是像我一樣放在 Github 上面都可以，你想怎麼玩就怎麼玩。

👨‍🏫總結

要達成目標，有很多方式可以實現，先分析自己想要的是什麼，再慢慢為自己找到合適的工具與做法，沒有一種方案是 100% 萬用的，上面的想法也是玩了幾次 Blog 以後( 從 FC2 -> Blogger -> 現在 )，從遇到的麻煩學取教訓，然後想辦法找到最適合自己的方案，再從這個方案動手改造成自己最順手的模式。

雖然過程會拉得比較長，但是自己做出來的那一刻，是非常有成就感的喔。🤠
